# 생기부 도우미

Claude API를 활용해 교사의 학생별 관찰 자료를 학교생활기록부(생기부) 서술형 문구로 변환해주는 웹서비스입니다.
승인제 회원가입, 30일 롤링 사용량 한도, 민감정보 1차 차단, 관리자 승인/모니터링 화면을 포함한 MVP입니다.

## 기술 스택

- 백엔드/프론트: Python, FastAPI + Jinja2 서버 렌더링
- 인증/DB: Supabase (PostgreSQL + Supabase Auth, RLS)
- AI: Anthropic Claude API (`claude-sonnet-5` 기본)
- 배포: Fly.io (도쿄 `nrt` 리전)

## 1. Supabase 설정

1. [Supabase](https://supabase.com)에서 새 프로젝트를 생성합니다.
2. Supabase 대시보드 **SQL Editor**에서 `supabase/schema.sql` 내용을 그대로 실행합니다.
   - `profiles`(승인 상태/역할), `generations`(생성 이력) 테이블과 RLS 정책, 신규 가입 시 프로필 자동 생성 트리거가 생성됩니다.
3. **Authentication > Providers**에서 이메일 인증(Confirm email) 사용 여부를 프로젝트 정책에 맞게 설정합니다.
   - 이메일 인증을 켜두면 사용자는 이메일 인증 후에도 관리자 승인이 있어야 생성 기능을 사용할 수 있습니다(이중 게이트).
4. **Project Settings > API**에서 `Project URL`, `anon public key`, `service_role key`를 확인해 둡니다. `service_role key`는 절대 클라이언트/저장소에 노출하지 마세요.

### 최초 관리자 지정

일반 가입 절차로 관리자 계정도 먼저 가입한 뒤, SQL Editor에서 아래 쿼리로 승격합니다.

```sql
update public.profiles
set role = 'admin', status = 'approved', approved_at = now()
where email = 'your-admin-email@example.com';
```

## 2. 환경 변수

```bash
cp .env.example .env
```

`.env` 파일을 열어 다음 값을 채웁니다.

| 변수 | 설명 |
| --- | --- |
| `SUPABASE_URL` / `SUPABASE_ANON_KEY` / `SUPABASE_SERVICE_ROLE_KEY` | Supabase 프로젝트 설정값 |
| `ANTHROPIC_API_KEY` | 본 프로젝트 전용으로 발급한 Anthropic API 키 (관리자 개인 키 1개, 타 프로젝트와 공유 금지) |
| `ANTHROPIC_MODEL` | 기본 `claude-sonnet-5`, 필요 시 Opus/Haiku로 전환 |
| `SESSION_SECRET` | 세션 쿠키 서명용 임의의 긴 문자열 |
| `MONTHLY_LIMIT` / `ROLLING_WINDOW_DAYS` | 사용량 한도 정책 (기본 30일당 200건) |

**비용 안전장치**: [Anthropic 콘솔](https://console.anthropic.com)에서 Spending Limit을 월 15만원으로 설정해 두세요. 이 값은 코드가 아닌 Anthropic 콘솔에서 직접 설정합니다.

## 3. 로컬 실행

```bash
pip install -r requirements.txt
uvicorn app.main:app --reload --port 8080
```

`http://localhost:8080` 접속 → 회원가입 → (SQL로 관리자 승인 또는 관리자 계정으로 로그인 후 `/admin`에서 승인) → 로그인 → 생성 기능 확인.

## 4. Fly.io 배포 (도쿄 리전)

### 최초 1회 설정

```bash
fly launch --no-deploy   # fly.toml이 이미 있으므로 기존 설정을 사용하도록 선택
fly secrets set \
  SUPABASE_URL=... \
  SUPABASE_ANON_KEY=... \
  SUPABASE_SERVICE_ROLE_KEY=... \
  ANTHROPIC_API_KEY=... \
  SESSION_SECRET=...
```

`fly secrets set`으로 등록한 값은 Fly.io 서버에 암호화되어 저장되며, 이후 배포(`fly deploy` 또는 GitHub Actions)에서 계속 유지되므로 매번 다시 등록할 필요가 없습니다. `fly.toml`은 `primary_region = "nrt"`(도쿄), 헬스체크 경로 `/healthz`가 이미 설정되어 있습니다. 앱 이름이 이미 사용 중이면 `fly.toml`의 `app` 값을 원하는 이름으로 변경하세요. `min_machines_running = 1`로 설정되어 있어 트래픽이 없어도 머신 1대가 항상 켜져 있습니다(콜드 스타트로 인한 첫 로딩 지연 방지). 비용을 아끼고 싶다면 `0`으로 바꿀 수 있지만, 그러면 한동안 접속이 없다가 처음 들어오는 사용자는 머신이 재부팅되는 동안(수 초) 로딩이 느려집니다.

### GitHub Actions 자동 배포

`main` 브랜치(또는 저장소 기본 브랜치)에 push할 때마다 `.github/workflows/fly-deploy.yml` 워크플로우가 자동으로 `fly deploy`를 실행합니다.

1. Fly.io 배포 토큰 발급: `fly tokens create deploy -x 999999h`
2. GitHub 저장소 **Settings > Secrets and variables > Actions**에서 `FLY_API_TOKEN`이라는 이름으로 위 토큰 값을 등록합니다. (앱 자체의 `SUPABASE_*`, `ANTHROPIC_API_KEY` 등은 GitHub이 아니라 위 `fly secrets set`으로만 등록합니다.)
3. 이후 브랜치에 push하면 자동으로 배포됩니다. 저장소 기본 브랜치 이름이 `main`이 아니라면 `.github/workflows/fly-deploy.yml`의 `branches:` 값을 실제 기본 브랜치명으로 맞춰주세요.

수동 배포만 원한다면 워크플로우 파일을 삭제하고 로컬에서 `fly deploy`만 실행해도 됩니다.

## 5. 정책 요약

- **가입 제한**: 이메일 도메인이 한국 시도교육청 도메인(`app/email_domains.py`의 목록, 예: 인천 `@ice.go.kr`)이거나 그 학교 서브도메인인 경우에만 가입할 수 있습니다. 목록이 실제와 다르면 해당 파일만 수정하면 됩니다. 회원가입 화면에서도 이메일을 "아이디" + "@도메인" 선택 상자로 입력받아 도메인을 직접 타이핑하지 않아도 되며(맨 앞은 인천 `ice.go.kr` 고정), 로그인 화면과 달리 어차피 서버가 이 목록 밖의 도메인은 거부하므로 "직접 입력" 옵션은 제공하지 않습니다.
- **로그인 이메일 입력 편의**: 로그인 화면에서 이메일을 "아이디" + "@도메인" 선택 상자로 나눠 입력받습니다. 도메인 목록은 `app/email_domains.py`의 교육청 도메인(이미 공개된 정보)만 후보로 보여주며, 가입자 이메일 목록 자체는 비인증 상태에서 절대 노출하지 않습니다(피싱 등 악용 방지). 마지막으로 선택한 도메인은 브라우저 `localStorage`에 저장되어 다음 로그인부터 자동 선택됩니다. 교육청 도메인이 아닌 이메일(관리자 등)로 로그인해야 하면 도메인 선택 상자에서 "직접 입력"을 고르면 이메일 전체를 직접 입력할 수 있습니다.
- **승인 플로우**: 가입 즉시 로그인은 가능하나 생성 기능은 차단되고 대기 화면이 표시됩니다. 관리자가 `/admin`에서 승인/거절합니다.
- **로그인 세션 유지/자동 갱신**: 브라우저 세션 쿠키 자체는 오래 유지되지만, 그 안에 담긴 Supabase 인증 토큰(access_token)은 기본 1시간 후 만료됩니다. 매 요청마다 거치는 로그인 확인 로직(`app/deps.py`의 `get_current_user`)이 토큰 만료를 감지하면, 로그인 시 함께 저장해 둔 refresh_token으로 조용히 갱신을 시도한 뒤 요청을 이어서 처리합니다. 갱신도 실패하는 경우(예: 다른 기기에서 로그아웃했거나 refresh_token 자체가 무효화된 경우)에만 실제로 로그아웃되고 로그인 화면으로 이동합니다. 이 갱신 로직이 없으면, 폼을 오래 작성하다 제출하는 순간 갑자기 로그아웃되며 입력하던 내용이 그대로 날아가는 문제가 있었습니다.
- **사용 한도**: 1인당 가입일 기준 30일 롤링 윈도우당 기본 200건(`MONTHLY_LIMIT` 환경변수, 최초 생성 + 재생성/수정 요청 모두 포함). 초과 시 다음 리셋일까지 완전 차단됩니다. 관리자(role=admin) 계정은 한도가 적용되지 않습니다. `/admin` 화면에서 관리자가 교사별로 한도를 개별 지정할 수 있으며(`usage_ledger.monthly_limit`), 비워두고 저장하면 전역 기본값으로 되돌아갑니다.
- **민감정보**: 주민등록번호 패턴은 정규식으로 자동 차단됩니다. 그 외 민감정보(가족관계/건강/상담 등)는 자동 탐지 없이 안내 문구로만 고지합니다.
- **과목/성취기준**: 과목을 선택하면 그 과목의 성취기준 코드(예: `12정01-01`) 목록이 활동별 선택 상자에 나타납니다. 활동 1개에 성취기준을 여러 개(복수 선택) 고를 수 있습니다. 과목·성취기준 데이터는 `app/subject_criteria.py`의 `SUBJECT_CRITERIA`에서 관리하며, 실제 교육과정 성취기준으로 교체해서 사용해야 합니다(현재 값은 예시입니다). 학업 성취도는 A~E 중에서 선택합니다. 같은 성취기준을 서로 다른 활동에 중복 선택하는 것은 여전히 허용되지 않습니다(활동 안에서 여러 개를 고르는 것과, 여러 활동에 같은 걸 중복으로 고르는 것은 다릅니다).
- **교사별 담당 과목**: 회원가입 화면에서 그때까지 등록된 과목(`SUBJECT_CRITERIA`의 전체 목록) 중 본인이 담당하는 과목을 복수 선택할 수 있고, `/account`(계정 관리)에서 나중에 언제든 바꿀 수 있습니다. 대시보드의 "과목" 선택 상자에는 본인이 등록한 과목만 나타나 다른 교사의 과목과 섞이지 않습니다. 아무 과목도 등록하지 않은 계정(과거 가입자 포함)은 하위 호환을 위해 전체 과목이 대신 표시됩니다. `/generate` 서버 검증에서도 본인이 등록한 과목 외의 값은 거부합니다.
- **학생 일괄 처리**: 화면에서 "+ 학생 추가"로 학생 블록을 늘려 한 번에 여러 명(기본 최대 20명, `app/generation.py`의 `MAX_STUDENTS_PER_BATCH`에서 조정 가능)을 순차적으로 생성할 수 있습니다. 사용 한도는 학생 수만큼 미리 확인하며, 부족하면 요청 전체를 거절합니다. 학생별로 생성이 실패해도 나머지 학생은 계속 처리되고, 실패한 학생만 결과 화면에 오류로 표시됩니다. 학번/과목/활동이 비어 있는 등 조건을 만족하지 못해 아예 생성이 시도되지 않은 학생도 어떤 학생이 왜 제외됐는지 화면에 표시됩니다. 학생이 많아져 화면이 길어지는 것을 막기 위해 학생 블록마다 접기/펼치기가 가능하고, "모두 접기"/"모두 펼치기" 버튼으로 한 번에 전환할 수 있습니다. "일괄 생성하기"를 누르면 처리 중임을 알 수 있도록 버튼이 비활성화되고 "생성 중입니다..." 문구로 바뀝니다.
- **재생성/중복 경고**: 같은 학번을 다시 생성해도 기존 생성 이력을 덮어쓰지 않고 새 이력으로 추가되며, 이미 이력이 있는 학번을 다시 생성하면 결과 화면에 경고가 표시됩니다. "일괄 생성하기"를 누르는 시점에도 브라우저에서 먼저 확인해, 이미 이력이 있는 학번(같은 배치 안에서 중복 입력된 경우 포함)이 있으면 "기존 이력을 덮어쓰지 않고 새 결과로 추가된다"는 확인창을 띄우고 교사가 승인해야만 실제로 생성 요청을 보냅니다.
- **입력값 즉시 검증**: "일괄 생성하기"를 누르면 서버로 보내기 전에 브라우저에서 먼저 학번 미입력/과목 미선택/활동 미입력/성취기준 중복 선택 여부를 확인합니다. 문제가 있으면 요청을 보내지 않고 어떤 학생에게 어떤 문제가 있는지 바로 화면에 표시하며, 문제 있는 학생 블록이 접혀 있었다면 자동으로 펼쳐집니다(브라우저 자체 필수입력 검사는 꺼 두어, 접힌 블록 때문에 아무 반응 없이 조용히 막히는 문제를 방지). 서버 쪽에서도 동일한 조건을 다시 검증하므로 자바스크립트가 꺼져 있어도 안전합니다.
- **임시저장**: 입력 중인 학생 목록과 목표 바이트는 "임시저장" 버튼으로 서버(`drafts` 테이블)에 저장할 수 있습니다. 사용자당 최근 1건만 보관(덮어쓰기)되며, 로그인 후 대시보드에 접속하면 자동으로 마지막 임시저장 내용이 불러와집니다. 유효성 검사 없이 입력한 그대로 저장되므로 작성 도중에 저장해도 됩니다.
- **활동 개수**: 학생별로 활동은 1개로 고정되지 않고 화면에서 자유롭게 추가/삭제할 수 있습니다(최대 10개, `app/generation.py`의 `MAX_ACTIVITIES`에서 조정 가능).
- **생성 결과**: 입력한 교과 성취 수준과 활동들을 모두 종합해, "교과 성취 수준 → 수행 특기사항 → 교과 역량 → 수업 태도" 순서를 따르는 **하나의 세특 문단**으로 생성합니다(활동별로 나누지 않음). 교과 성취 수준(A~E)은 문장의 깊이·어조를 정하는 참고 자료로만 쓰이며, "A 수준"·"성취도가 우수함"처럼 등급을 문장에 직접 언급하지 않고 학습 내용·성장 서술에 자연스럽게 녹여 씁니다.
- **바이트**: 목표 바이트(최소/최대)는 화면에서 교사가 직접 지정합니다(기본값 600~700바이트). 최대 바이트는 2500바이트를 넘을 수 없으며, 이를 넘는 값을 입력하면 조용히 2500으로 낮추지 않고 오류로 처리해 다시 입력하도록 합니다(`app/generation.py`의 `HARD_MAX_CHAR_LIMIT`/`_clamp_char_limits()`). 바이트 수는 나이스(NEIS) 공식 계산식(`=2*LENB(cell)-LEN(SUBSTITUTE(cell,CHAR(10),""))`)을 그대로 이식한 `app/charcount.py`의 `neis_byte_count()`로 계산해 화면에 표시합니다. 생성 결과가 목표 바이트 범위를 벗어나면, 같은 대화에서 분량만 조정해 다시 작성하도록 최대 2회 재요청합니다(`app/generation.py`의 `MAX_LENGTH_RETRIES`). 단, 목표 최대 바이트의 5%(`LENGTH_TOLERANCE_RATIO`) 이내로 근소하게 벗어난 경우는 재시도 없이 그대로 받아들입니다(`_is_length_acceptable()`) — 재시도 1회마다 API 호출 비용이 그대로 늘어나므로, 목표 바이트가 클수록(예: 1500~1800바이트) 근소한 오차까지 매번 다시 쓰게 하면 비용 대비 이득이 적기 때문입니다.
- **데이터 보관**: 생성 결과는 만료 없이 영구 보관되며, 사용자가 계정 삭제를 요청하면 즉시 하드 삭제(생성 이력 + 프로필 + Supabase Auth 계정)됩니다. 관리자 계정은 삭제할 수 없습니다. 같은 이메일로 재가입해 사용 한도를 우회하는 것을 막기 위해 **이메일 단위 사용 건수/주기 기록(`usage_ledger`)만은 삭제되지 않고 남습니다** (생성 문구 등 실제 개인정보 내용은 포함하지 않음).
- **RLS**: `profiles`/`generations`/`drafts` 모두 본인 행만 조회·수정 가능하도록 RLS가 설정되어 있습니다. 관리자 승인/모니터링은 서버가 `service_role` 키로 RLS를 우회해 처리하며, 이 키는 서버 환경변수로만 보관됩니다.
- **동시 접속 처리**: Supabase(`supabase-py`)와 Anthropic SDK는 모두 동기(블로킹) 클라이언트입니다. FastAPI 라우트는 전부 `async def`인데, 이 블로킹 호출을 그대로 실행하면 응답을 기다리는 동안 이벤트 루프 전체가 멈춰 다른 사용자의 요청까지 함께 지연됩니다(특히 `/generate`는 학생 1명당 Claude API를 여러 번 순차 호출하므로 영향이 큼). 모든 블로킹 호출은 `starlette.concurrency.run_in_threadpool`로 감싸 스레드풀에서 실행하므로, 한 사용자가 생성 요청 중이어도 다른 사용자의 페이지 조회/생성 요청이 함께 처리됩니다.
- **화면 구성**: 로그인하면 왼쪽에 사이드바가 표시되며, "생기부 문구 생성"(`/dashboard`), "생성 이력"(`/history`), "성취기준 조회"(`/criteria`), "계정 관리"(`/account`, 담당 과목 수정 + 계정 삭제, 관리자 제외), "관리자 모드"(`/admin`, 관리자만)로 이동할 수 있습니다. 승인 대기 중에는 생성/이력/성취기준 조회 메뉴가 숨겨지고 계정 관리만 노출됩니다.
- **성취기준 조회**: `/criteria`에서 과목을 선택하면 그 과목에 등록된 성취기준 코드/설명 전체를 표로 볼 수 있고, 코드나 설명 텍스트로 검색할 수 있습니다. 대시보드와 달리 본인이 등록한 과목뿐 아니라 시스템에 등록된 모든 과목을 조회할 수 있어, 새로 담당할 과목을 고르기 전에 미리 훑어보는 용도로도 쓸 수 있습니다.
- **생성 이력 검색/과목 필터/정렬/페이지네이션**: `/history`에서 학번·영역·결과 내용을 대상으로 검색할 수 있고(대소문자 구분 없음), "과목" 드롭다운으로 특정 과목의 이력만 골라볼 수 있습니다(다른 과목과 섞이지 않도록). "일시"/"학번"/"영역" 열 제목을 클릭하면 해당 기준으로 오름차순/내림차순 정렬됩니다(다시 클릭하면 순서가 뒤집힘). "페이지당 표시"에서 10/20/100명씩 보기를 선택할 수 있고, 하단에서 페이지를 넘길 수 있습니다(`app/generation.py`의 `HISTORY_PAGE_SIZES`에서 조정 가능). 검색/필터/정렬/페이지네이션은 최근 5000건(`app/generation.py`의 `HISTORY_FETCH_LIMIT`) 범위 내에서 서버가 메모리상에서 처리합니다 — 교사 한 명의 실사용 규모를 여유 있게 덮는 수준이며, 이 건수를 넘는 아주 오래된 이력은 검색/필터에 걸리지 않습니다.
